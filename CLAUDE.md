# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

General Hebrew math practice app ("קסם החשבון") for elementary grades **גן through כיתה ו** (kindergarten–6th), following the Israeli Ministry of Education curriculum. Each kid **registers** (name + grade) and takes a short **placement test** that sets their starting level; the adaptive engine then raises/lowers difficulty by performance. Multiple kids live on one device. Plain HTML/CSS/JS with no frameworks or build step: `index.html` (markup + all JS), `style.css`, `assets/` (images + fonts), plus PWA files (`manifest.webmanifest`, `sw.js`, `icon-*.png`, `favicon.svg`). Works offline from `file://`; no network dependencies at runtime. All UI text is Hebrew, feminine address forms ("נסי", "לחצי").

(History: started as a fixed-profile app for three specific kids — noa/raz/tzuri — whose tracks became the גן/א/ג curricula. Those hardcoded profiles are gone; everything is now keyed by grade + a dynamic user list.)

## Commands

- **Run tests:** `node _logic_test.js` — extracts the pure-logic section out of `index.html` and validates exercise generation (thousands of exercises per type: no negative subtraction results, no division remainders, choice options contain the answer exactly once) plus the adaptive level/reinforce state machine. Its migration test also executes the **entire** `<script>` under Node with browser stubs, so it doubles as a syntax check for the DOM code. Run after any change to `index.html`.
- **Visually verify new exercise visuals** (SVG, hints): generate a temp gallery page from the extracted logic (small Node script that calls `genExercise` / the SVG helpers `angleSVG`/`shapeSVG`/`clockSVG`/`rectSVG`/`triSVG` and writes sample HTML), screenshot it with headless Edge, then delete the temp files. Deep levels can't be reached via the URL hash, so this is the way to see late-level content without playing through. NB: when eval'ing the extracted logic at module scope, wrap it in an IIFE (`(function(){ return eval(code + ';({genExercise})'); })()`) — a bare top-level `eval` leaks the `const`/`function` declarations and clashes with your own `const { genExercise }`.
- **Open the app:** `Start-Process index.html`, or in a browser at https://dyfroman.github.io/math-app/.
- **Render smoke test:** headless Edge screenshot, e.g.
  `& "C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe" --headless --disable-gpu --window-size=480,860 --virtual-time-budget=4000 --screenshot=out.png "file:///C:/Users/dyfro/Desktop/math-app/index.html#register"`
  The URL hash now deep-links to `#register` (the registration screen) or `#<userId>` (an already-registered kid's game screen — ids are generated, so this is only useful after seeding). The old `#noa`/`#raz`/`#tzuri` deep-links are **gone**. **Always pass `--virtual-time-budget=4000`:** without it, screenshots freeze CSS animations at an early frame (e.g. the card's entry animation makes it look half-transparent) and you'll chase phantom styling bugs. Headless Edge under PowerShell exits non-zero because of the stderr-redirect quirk even when the PNG is written fine — check the file exists rather than trusting the exit code, and run each screenshot from its own `--user-data-dir` to avoid the singleton-profile lock.
- **Screenshot states you can't deep-link to** (game mid-track, journey map, celebrate, parent dashboard, home with several kids): write a tiny Node script that copies `index.html` to a temp file and patches the load handler to **seed users** (`saveUsers([{id,name,gradeId,themeKey,emoji}])` + a per-user `localStorage.setItem(storageKey(id), JSON.stringify(state))`) and route on a custom hash, e.g. `if (h==='demo-journey'){ startChild('d1'); openJourney(); return; }` or `if (h==='demo-parents'){ renderParents(); return; }`. Note `location.hash` includes the leading `#` — strip it (or compare against the already-stripped `h`) before matching. Then screenshot the temp file and delete it.
- **Deploy:** push to `main`. GitHub Pages (repo `dyfroman/math-app`) serves the branch root automatically. `gh` CLI is **not** installed; GitHub API calls work via the token from `git credential fill`.
- **Hebrew commit messages:** avoid ASCII double quotes (`"`) inside the message — even in a single-quoted PowerShell here-string, embedded `"` breaks native argument quoting for `git commit -m` (Windows PowerShell 5.1) and the text after the quote is parsed as a pathspec. Rephrase or use Hebrew gershayim (״) instead.

## Architecture

The app is `index.html` (markup + JS) plus `style.css` (all styling) and `assets/` (theme images + self-hosted Rubik woff2 fonts — total ~750KB, keep backgrounds ≤400KB JPG and mascots ≤200KB PNG). If asset files are missing the app degrades gracefully (gradient backgrounds, no mascot via `onerror`, system font).

**PWA:** `manifest.webmanifest` (standalone, rtl, theme/background colors) + `sw.js` (app-shell cache for offline + installability) + `icon-192/512.png` + `favicon.svg`, all linked from `<head>`. The SW caches by a `VERSION` constant — **bump `VERSION` in `sw.js` whenever you change `index.html`/`style.css`** or installed users stay on the stale cache. Registration is guarded (`navigator.serviceWorker` + `http(s)`) so it no-ops on `file://` and under the Node test stubs.

**Users & grades (the core data model):** `CONFIG.grades` (keyed `gan,g1,g2,g3,g4,g5,g6`) holds each grade's `{label, short, input, trackVersion, startLevel, levels:[...]}` curriculum track. Registered kids are a dynamic list in `localStorage['math_app_users']` (`[{id,name,gradeId,themeKey,emoji}]`), each with progress under `math_app_user_<id>`. `profileOf(user)` merges a user with their grade track into the runtime `curProfile` (set in `startChild`); `GRADE_ORDER` controls the registration grade picker (filtered to grades that actually exist, so adding a grade auto-appears). Registration (`openRegister`/`submitRegister`) → placement test (`startPlacement`/`renderPlacement`/`finishPlacement`, scored by the pure `placementStartLevel`) → game.

**Regenerating assets** (Canva + Adobe MCP connectors): backgrounds are Canva `generate-design` with `design_type: phone_wallpaper` (prompt must demand "no text, empty calm middle" so cards stay readable), exported as JPG ~1700px; mascots are `instagram_post` sticker-style characters on a plain white background. Generated candidates often sneak in text/frames — export and eyeball each one, try the next candidate if bad. Canva's "transparent" PNG export keeps the white painted inside the illustration, so run Adobe `image_remove_background` on every mascot; Adobe rejects `export-download.canva.com` URLs (domain not whitelisted), so first upload the local file via `asset_initialize_file_upload` → curl PUT (`-T file`, one call per file — parallel `--data-binary @file` jobs fail in git-bash) → `asset_finalize_file_upload`, and pass the resulting presigned URL. Downscale mascots to 400×400 (PowerShell System.Drawing works) before committing.

**Theming:** three adventure worlds — `space` (מסע בין כוכבים), `forest` (יער הקסמים), `farm` (חוות החיות). The `THEMES` object (DOM section of `index.html`) is keyed by **world key** (`space`/`forest`/`farm`, matching the `body[data-theme="…"]` selectors in `style.css`) → world name, `color`, background image, mascot poses (happy/cheer PNGs), journey-map node icons, and praise strings. A world is assigned randomly at registration (`THEME_KEYS`, stored on the user as `themeKey`); the avatar emoji comes from `AVATAR_EMOJIS`. `applyTheme(themeKey)` sets `body[data-theme]` and the `--accent` fallback; all colors flow through CSS variables (`--accent`, `--accent-soft`, `--accent-ink`, `--key-shadow`, `--page-ink`, …) defined per theme in `style.css`. The background image sits on a fixed `body::before` layer (not `background-attachment:fixed`, which is broken on iOS); the body gradient behind it is the no-image fallback.

**Journey map:** `#screen-journey` + `renderJourney()` show a vertical zigzag path of the profile's levels (column-reverse — start at bottom), with done/current/locked node states and the mascot perched on the current node. A mini trail (`renderTrail()`) replaces the plain level text in the game HUD. On level-up the map auto-opens (~2.3s after the banner) with a `just-advanced` node animation. The mascot (`#mascot`, `setMascot(pose)`) reacts in `onCorrect`/`onWrong`/`nextExercise` and stars on the celebrate screen.

The JS in `index.html` is one `<script>` block, split in two by comment markers:

1. **Pure logic** between `==LOGIC-START==` and `==LOGIC-END==` — `CONFIG` (incl. `grades`), `TYPE_NAMES`, `GRADE_ORDER`, `genExercise(type, easy)`, the SVG helpers, `updateProgress(...)`, `chooseType(profile, st)`, `defaultState(grade)`, `placementStartLevel(correct, total, levelsCount)`. This section must stay free of `document`/`window`/`localStorage` references: `_logic_test.js` extracts it by those markers and runs it under Node via `eval`. New helper functions used by generators belong here. (`THEMES`/`AVATAR_EMOJIS`/`THEME_KEYS` and the user model live below the marker, in the DOM section.)
2. **DOM/interaction code** below the end marker — screens, rendering, sounds (Web Audio, generated in code — no audio files), effects, parent dashboard.

Key design points that span the file:

- **`CONFIG.grades` drives everything.** Each grade has `levels: [{title, types: [...]}]`, plus `label`/`short`/`input`/`trackVersion`/`startLevel`; each type string maps to a `case` in `genExercise` and a label in `TYPE_NAMES`. Adding a level or exercise type means touching those three places only (and bumping the grade's `trackVersion` if the levels array is restructured — see migration below). Generators are **shared across grades** by curriculum fit (e.g. `addsub100`/`place_value` appear in several grade tracks); type ids still differ where the output format differs (`add10` numpad vs `tadd10` emoji-choice). Grade tracks mirror the Israeli MoE curriculum per grade (גן counting/shapes → ו ratio/percent/algebra/volume); verify topics against the official אבני דרך at pop.education.gov.il when extending a grade.
- **Exercise object contract:** `{type, mode: 'numpad'|'choice', prompt, line?, visual?, answer, options?, hint, emojiLine?, meta?}`. `mode` decides the input UI: גן uses only `choice` (big tap targets, no typing); other grades mostly `numpad`. **`choice` with string-valued answers** is the pattern for anything non-integer — geometry names, fractions (`'3/5'`), decimals (`'0.7'`), clock times (`'3:30'`); `submitAnswer` compares with `===`, which works for both. In `numpad` mode the `?` in `line` is replaced by the answer box. **A `line` containing `=` is auto-evaluated by the test**, so any HTML in a line (e.g. a `dir="ltr"` attribute — the `=` counts!) or a non-integer/algebraic value breaks it: put fraction/decimal/equation expressions in `visual` (innerHTML, not evaluated), not `line`. `hint` is an HTML string shown after the second wrong attempt; after 4 attempts the answer is revealed. `meta` carries generator internals consumed only by the tests. Numbers ≥1000 use comma separators via `fmt()`.
- **Adaptive flow** (`updateProgress`): 5 first-try corrects in a row → level up; 3 wrong-first-try in a row on one type → "reinforce" mode (`st.reinforce = {type, left}`) which serves easy variants of that type until enough corrects; no level-ups while reinforcing. Stats count each exercise once: `c` if solved on the first try, `w` otherwise. The **placement test** (post-registration) seeds the starting `st.level` via `placementStartLevel` (proportional to score, capped one below the final "מבחן מסכם" level); the adaptive engine takes over from there.
- **Persistence:** the user list is `localStorage['math_app_users']`; each kid's full state (`defaultState(grade)`) is under `math_app_user_<id>` (`storageKey(id)`). `loadState(id)` merges over defaults so adding new state fields is backward-compatible. Sessions are 10 exercises; `session.done` persists so a kid resumes mid-session. **Track migration:** if a saved `trackVersion` differs from the grade's, `loadState` keeps stars/bestStreak/stats but resets level to `startLevel` and clears session/reinforce. (Note `parsed.trackVersion || 1` treats `0` as "unset" — use version numbers ≥1.)
- **RTL/LTR:** the document is `dir="rtl"`, but arithmetic lines render inside `.exline` which forces `direction:ltr` so `3 + 4 = ?` reads left-to-right. The minus sign is `−` (U+2212), multiplication `×`, division `÷` — the test validates any `line` containing `=` by substituting the answer for `?`, converting those symbols, and evaluating both sides (handles order-of-operations lines with parentheses too). Inline math inside Hebrew hint text is wrapped in `<span dir="ltr">`.
- **Parent screen** is gated by a multiplication question (`CONFIG.parentQuestions`) and iterates `math_app_users`, reading each kid's stats from localStorage; per kid there's a two-tap "איפוס התקדמות" (clears state) and "מחיקת ילד" (removes from the user list + deletes state).
